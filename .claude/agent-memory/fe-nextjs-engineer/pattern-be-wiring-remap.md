E18 epic swaps mock-first repos → real edu-api contract. Recurring remap shape (US-E18.1 calendar, mirrors US-E18.0 school-config):

**Why:** mock-first repos were written against *guessed* wire shapes; the real `edu-api/services/<svc>/docs/openapi.yaml` + `ERROR_CODES.md` are the contract authority. Fixes stay at infra layer (DTO/mapper/repo/error-map) — domain entity + presentation VM contract must NOT change (zero UI regression is the epic AC).

**How to apply:**
- **Flat wire → nested entity.** BE returns separate resources (e.g. year has no nested `terms[]`). Keep the domain entity nested; split the mapper: `fromFlatDto(dto): Omit<Entity,"children">` + `fromDto(dto, childDtos[]): Entity`. Repo reassembles by fanning out to `listChildren(parentId)` per parent via `Promise.all` inside the try (raw http throws unmapped ApiError → outer catch maps it).
- **status enum → boolean.** `isActive = status === "ACTIVE"`. Filter soft-deleted (`status === "ARCHIVED"`) out of list to preserve mock's "disappears" UX (BE doesn't hard-delete).
- **Non-atomic orchestration** (create-then-activate): do the POST inside try (maps its own errors), then call the second op (`activateYear`) OUTSIDE the try so its already-mapped `CalendarFailure` isn't double-mapped by the outer `mapCalendarFailure(err)` (which would return `unknown` — errorCodeOf on a failure object is undefined). Don't roll back if BE has no delete endpoint.
- **Full error matrix.** Cover EVERY `<PREFIX>_*` code in ERROR_CODES.md, branch on `error.code` never message. Watch for pre-existing WRONG mappings (calendar had `CALENDAR_FORBIDDEN`→network-error). Defensive-map id-format codes (they come from prior API responses, not user input) to nearest not-found/forbidden, never fall to `unknown`.
- **Proactive refresh gap.** Every real (`!USE_MOCK`) DI factory must `await ensureFreshSession()` (from `@/bootstrap/di/auth.di`) before `createServerHttpClient()`. Documented in decision 0018 but historically only wired in auth.di — each wiring US closes it for its own cluster. Copy comment style from `admin-school-setup.di.ts`.
- **Mock repo** operates on domain entities → unaffected by DTO remap; just add a NOTE comment documenting real-path semantic divergences (soft-delete, no auto-swap) so readers don't assume parity.
- **Error rendering** in calendar-screen is generic `tErrors(errorKey)` where `errorKey: Failure["type"]` — adding union members + i18n keys (vi+en) is enough; tsc enforces coverage, no switch case. Verify the screen isn't doing an exhaustive switch before assuming. NOTE: staffing duplicates the SAME `errors.*` namespace under each screen (departments/positionTitles/assignments) — add every new failure key to ALL of them in vi+en.

**US-E18.2 staffing add-ons (confirmed sub-patterns for fields BE doesn't return):**
- **id rename is near-universal.** Wire uses `<entity>Id` not `id` (departmentId/positionTitleId/positionAssignmentId, like calendar's academicYearId). Also watch timestamp renames (`createdAt`→domain `assignedAt`). Always diff the DTO field-by-field against openapi even when the audit says "MATCH".
- **Derived count (not on wire).** When a list badges an interpolated count (e.g. `activeAssignmentCount`), prefer real derivation over `0`-fallback: private helper fully paginates the source list (`?status=ACTIVE` cursor loop until `!hasMore`), builds count Maps keyed by the join id, `Promise.all` the count-fetch alongside the main list fetch inside the try. Apply to single get/patch too so use-case guards read a real count. `0`-fallback is the documented alternative (US-E18.1 hasGrades precedent) — say which you took.
- **Join name from a sibling list.** `positionTitleName`-style fields: fetch the sibling list, build `id→name` Map, join. Missing entry → fall back to the id string.
- **No-source name gap.** IAM `MemberResponse` has NO name field and no by-id user lookup (only `/users/me` self) → `memberName` falls back to raw `memberId` with a comment; file cross-repo ask. Do NOT hide the column.
- **Derive a required-on-wire field from a related resource.** BE required `scopeEntityType` absent from web input → repo does `GET /position-titles/{id}` to read `scopeType`, sends it in the POST body (and reuses the fetched entity for the name join). Keeps it repo-only, no UI/domain field.
- **Mapper takes derived extras as params.** `toDepartment(dto, count=0)`, `toPositionAssignment(dto, {memberName, positionTitleName})` — the repo threads in what the wire lacks.

**US-E18.3 subject-catalogue add-ons:**
- **Split concept field** (same as staffing dept): wire `conceptLabelSuggested` (enum, nullable) + `conceptLabelCustom` → domain single `conceptType`. Request builders map back: `conceptType`→`conceptLabelSuggested`, omit when null. BE also computes `effectiveConceptLabel` — do NOT consume it (presentation recomputes locally).
- **Nested `master` block + ResourceRef.** Subject wire nests master fields under `master: MasterFields` (may be absent → default null-num/""-str). `learningOutcomes`→domain `outcomeTargets`, `requiredExamCount`→`requiredAssessmentCount`. `exerciseBankRef`/`examBankRef` are `ResourceRef {type,ref}` objects: read `.ref ?? ""` in; on write wrap `{type:"OPAQUE", ref}` when non-empty, **omit when ""** (don't send empty ResourceRef). Cleared (null) numbers are omitted (wire ints non-nullable → BE keeps prior value); empty strings ARE sent (clears).
- **No `parentId`/child filter on list → shared full fan-out + client filter.** `GET /subjects` has only status/cursor/limit. One private `fetchAllSubjectDtos()` (cursor loop, NO status filter → both ACTIVE+ARCHIVED, needed for counts AND the screen shows archived). `listParents` groups it by `subjectParentId` for childCount/activeChildCount; `listSubjects(parentId)` filters it client-side. (Known O(N) re-fetch when page also calls listSubjects per parent — logged cross-repo perf ask, not fixed in wiring US.)
- **Detail-panel enrichment with no reverse endpoint → return [].** `getSubject.classOfferings` (classes teaching subject X) has no BE reverse lookup → return `[]` always in real mode (UI already has empty state). Don't invent multi-fetch client-side assembly.
- **Method wired to a real screen action but NO BE endpoint (restore/un-archive) → WEB-ONLY optimistic `ok(undefined)`.** DON'T delegate to the mock repo: its method looks the entity up in the fixture seed and returns `not-found` for real UUIDs → breaks the flow. A bare `ok(undefined)` (no HTTP) lets the screen's optimistic flip apply; no server persistence (documented gap + cross-repo ask for the endpoint). This DEVIATED from the packet's "delegate to mock" suggestion — flag the reasoning to fe-lead.
- **Error-matrix corrections:** fix guessed codes (`INVALID_SUBJECT_CODE`→real `SUBJECT_INVALID_CODE`), add real ones (`SUBJECT_ARCHIVED`/`CLASS_SUBJECT_ARCHIVED`→new `subject-archived` failure+i18n), DROP guessed non-existent branches (`SUBJECT_PARENT_HAS_ACTIVE_CHILDREN`, `ROSTER_ACCESS_FORBIDDEN`). Always cross-check every branch against the real ERROR_CODES.md section.

**US-E18.14 fully-blocked force-mock, THROWING-repo variant (mirrors US-E18.8 staff-leave which was the Result-repo variant):**
- Some E18 features can't wire ANY op real → the whole feature stays mock-first: DI factory `makeRepo()` returns `new MockXxxRepository()` unconditionally (drop the `USE_MOCK`/`createServerHttpClient`/real-repo imports), documented why + story-packet path. Real repo class kept as permanent blocked stubs, still imported by its TEST file (not the DI). This is now the 3rd such (staff-leave, teaching-plan, discipline).
- **Stub shape depends on the repo's error convention** (check `i-<name>.repository.ts`): Result-repo (`Promise<Result<T>>`) → each method `return { ok:false, error:{type} }` (staff-leave). Throwing-repo (`Promise<Entity>`, throws Failure) → a `private blocked(): never { throw { type:"not-found" } satisfies XxxFailure }` and every method `return this.blocked()`. Keep `constructor(_http)` for signature parity with a `biome-ignore lint/complexity/noUselessConstructor` comment.
- **Stub guard tests:** for EVERY method assert `.rejects`/`ok:false` AND `expect(http.get/post/put/delete).not.toHaveBeenCalled()` — proves a stub can never silently hit the unwireable real API. Copy the makeHttp()+apiError() helper style from the precedent test.
- **toFailure kept correct + fully unit-tested even though unreachable** (for the day it unblocks): one assertion per ground-truthed code. Judgment-call reuses when no distinct UX: invalid-id/term-not-found→`not-found`, invalid-state→`invalid-transition`, violation invalid-input→`missing-description`, leave invalid-input→`reason-too-short`. Genuinely-new concepts get new union types + vi/en i18n keys (discipline added `same-actor`/`invalid-transition`/`locked`/`student-not-enrolled`). Put specific `case code:` branches BEFORE any generic `status===409/403/404` fallback (keep the legacy fallbacks for mock/back-compat).
- **Endpoint file remapped to real paths but "recorded, not consumed"** — document ops with no real equivalent inline (discipline: no hard `delete`, no conduct `override` [real is DRAFT→SUBMIT→APPROVE/REJECT], no parent-child classId discovery) rather than silently implying a false mapping.
